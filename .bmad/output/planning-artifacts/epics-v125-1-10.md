# V125-1-10 — `ArgoCDProvider` + per-type Test routing + UI badge — Epic Breakdown

## Overview

V125-1-10 is the implementation half of `docs/design/2026-05-13-cluster-connectivity-test-redesign.md`. It introduces a new `CredentialsProvider` that reads cluster credentials directly from the ArgoCD cluster Secret in the `argocd` namespace, dispatches Test by inspecting that Secret's `config` JSON shape, and ships a hostname-derived UI type badge. End result: Test cluster works on every dev install (kind / minikube / EC2-direct / static-token EKS) with zero extra configuration; cloud-IAM clusters surface specific actionable errors instead of the generic "no credentials provider configured" 503.

**Why now:** maintainer hit the bug live on 2026-05-14 ("i think test on the cluster is not working again"). Root cause traced to `internal/providers/k8s_secrets.go:119-138` — `KubernetesSecretProvider.GetCredentials` looks for a Secret in the `sharko` namespace named after the cluster with a `kubeconfig` data key, but the `register-cluster` flow only creates the ArgoCD-format Secret in the `argocd` namespace. Two stores, never bridged. `argocdProvider` is the bridge.

## Driving sources

- **`docs/design/2026-05-13-cluster-connectivity-test-redesign.md` §2.1-§2.4** — the four locked positions are this epic's scope contract. Every story acceptance criterion derives from these sections. Read it first.
- **`docs/design/2026-04-07-secrets-provider-design.md`** — the existing `SecretProvider` vs `ClusterCredentialsProvider` two-interface architecture this epic extends. `argocdProvider` implements the second; addon-secrets stay on Vault/AWS-SM/k8s.
- **`docs/design/2026-05-11-cluster-secret-reconciler-and-gitops-stance.md`** — articulates the "Sharko owns its solution end-to-end" stance and decision C "Sharko owns the cluster Secret it created." V125-1-10's read-only access to the ArgoCD Secret is consistent with that stance.

## Scope frame (verbatim from design doc §2)

> "Four locked positions, each addressing one layer of the redesign. They compose into a single coherent path: introspect the Secret to pick the auth method, render a hostname-derived badge for human recognition, and ship `argocdProvider` after V125-1-8 so ownership semantics are clean."

### Locked positions

1. **No stored cluster-type label** — runtime introspection only (§2.1). No `provider_type` / `displayType` / `clusterFlavor` field. Single source of truth = the ArgoCD cluster Secret's `config` JSON.
2. **Auth-method detection from ArgoCD Secret structure** (§2.2):
   - `bearerToken` + `caData`/`insecure` → ready-to-use, runs to completion (kind/minikube/EC2-direct/static-token EKS)
   - `awsAuthConfig` → IAM mode → specific actionable error pointing at IAM setup runbook
   - `execProviderConfig` → exec-plugin mode → specific error "not supported in v1.x, tracked for v2"
   - Errors carry stable identifier codes so UI renders type-specific copy + deep links.
3. **UI type badge** (§2.3) — hostname-derived at render time, never stored. Pills:
   - `*.eks.amazonaws.com` → orange `EKS`
   - `*.azmk8s.io` → blue `AKS`
   - `*.gke.io` / `*.googleapis.com` → red `GKE`
   - `kind-*` / `localhost` / `127.0.0.1` → gray `kind`
   - `*.minikube.io` → gray `minikube`
   - else → gray `Self-hosted` (or omit)
   - Custom-DNS misidentification accepted (cosmetic-only — Test routes off Secret structure, not the badge).
4. **`argocdProvider`** (§2.4) — new built-in `CredentialsProvider`, default for non-cloud auth. Reads from `argocd` namespace. Vault/AWS-SM/file-store providers stay (additive, not replacement).

### Explicitly out of scope (per design doc §3)

- EKS / AKS / GKE token-refresh flows (separate brainstorm — V125-1-10 surfaces actionable errors but doesn't implement cloud-creds plumbing)
- BUG-040 / BUG-042 (separate UI bundles)
- Storing cluster-type labels of any kind
- Migrating pre-V125-1-8 Sharko-created Secrets (covered by V125-1-8's one-shot label migration)
- Replacing existing Vault/AWS-SM providers

### What stays UNCHANGED (preservation contract — do not regress)

Per the 2026-04-07 two-interface architecture, V125-1-10 ONLY adds an implementation of `ClusterCredentialsProvider`. The orthogonal `SecretProvider` interface and the entire addon-secrets push flow stay exactly as they are today. Confirmed against maintainer 2026-05-14.

| Concern | Backend interface | V125-1-10 impact |
|---|---|---|
| **Addon secret VALUES** (Datadog API keys, GitHub tokens, etc.) — pushed to remote cluster K8s Secrets | `providers.SecretProvider.GetSecretValue` | ❌ NONE — entire flow preserved |
| **Cluster KUBECONFIGS** — used by Sharko to connect to clusters for Test, adoption, etc. | `providers.ClusterCredentialsProvider.GetCredentials` | ✅ ADDS `ArgoCDProvider` as new implementation, alongside existing ones |

Specifically preserved (V125-1-10 stories must NOT touch any of these):

- `internal/secrets/reconciler.go` — the 5-min addon-secrets reconciler that fetches values from the configured `SecretProvider` and pushes them to remote clusters via `remoteclient.EnsureSecret`
- `internal/providers/k8s_secrets.go::GetSecretValue` — Kubernetes-Secrets-backed `SecretProvider` for addon secret values
- `internal/providers/aws_sm.go` — AWS Secrets Manager `SecretProvider`
- `internal/providers/gcp_sm.go` — GCP Secret Manager `SecretProvider`

That **half** of the struct's surface gets a docs-only deprecation note in v1.25 (operators with corporate Vault adopt-flow installs depend on it; do not remove). Real removal is V125-2 cleanup. The `GetSecretValue` half stays untouched — addon secret values continue to resolve through whatever provider is configured.

## Sequence flip rationale (V125-1-10 → V125-1-9 → V125-1-8 → V125-1-7 tightening)

The locked design doc §4 originally placed V125-1-10 LAST (after V125-1-7 tightening) for cleanest ownership semantics — Sharko reads only Secrets it provably wrote post-V125-1-8.

**Maintainer reversed this 2026-05-14** ("i want you to fix it of course") because:

- V125-1-10 is **read-only** of the ArgoCD Secret. The ownership label V125-1-8 introduces does not affect a read-only consumer.
- The current `register-cluster` flow already creates the ArgoCD cluster Secret, so the data V125-1-10 reads exists today on every dev install — V125-1-10 has valid data to read against right now.
- Test cluster is broken on every fresh dev install in v1.24. Architectural ordering preference loses to "broken UX in production-class dev installs."
- Pre-V125-1-8 risk: V125-1-10 might read a Secret Sharko didn't create (e.g., in adopt scenarios). Mitigation: `argocdProvider` returns "not found" for missing Secrets, falls back gracefully — no different than today's behavior. Post-V125-1-8 the ownership label tightens this to "Sharko-managed only."

## Epic V125-1-10: `ArgoCDProvider` + per-type Test routing + UI badge

### Story 10.1: `ArgoCDProvider` implementation + auth-method detection

New file `internal/providers/argocd_provider.go` implementing `providers.ClusterCredentialsProvider` (interface at `internal/providers/provider.go:17-30`).

- Reads cluster Secrets from `argocd` namespace (default; configurable via `cfg.Namespace`).
- ArgoCD's cluster Secret format: labelled `argocd.argoproj.io/secret-type: cluster`, with data fields `name`, `server`, `config` (JSON blob).
- Parses the `config` JSON and routes per detected auth shape:
  * `bearerToken` (+ optional `tlsClientConfig.caData` / `tlsClientConfig.insecure`) → return `*Kubeconfig` with `Server` + `CAData` + `Token` populated. `Raw` is a synthesized minimal kubeconfig YAML (so existing CRUD-test flow that calls `clientcmd.RESTConfigFromKubeConfig` works).
  * `awsAuthConfig` present → return typed error `ErrArgoCDProviderIAMRequired` carrying error code `argocd_provider_iam_required` and a structured detail (cluster name, server URL).
  * `execProviderConfig` present → return typed error `ErrArgoCDProviderExecUnsupported` carrying error code `argocd_provider_exec_unsupported`.
  * Neither field present → return generic `ErrArgoCDProviderUnsupportedAuth` carrying error code `argocd_provider_unsupported_auth`.
- `ListClusters` returns all secrets matching the label selector `argocd.argoproj.io/secret-type=cluster`, decoded to `ClusterInfo` (Name from secret data `name`, Region from a derived label if present, otherwise empty).
- `SearchSecrets` returns secret names matching substring (parity with KubernetesSecretProvider).
- `HealthCheck` lists Secrets in the `argocd` namespace with the cluster-type selector (limit 1).

**Acceptance criteria:**

- Unit tests with `fake.NewSimpleClientset` (kubernetes/client-go testing pattern — same `kubernetes.Interface` shape used by `newKubernetesSecretProviderWithClient`):
  * `bearerToken` happy path returns `*Kubeconfig` with all fields populated
  * `bearerToken` + `insecure: true` (no `caData`) works
  * `awsAuthConfig` returns `ErrArgoCDProviderIAMRequired` with structured detail
  * `execProviderConfig` returns `ErrArgoCDProviderExecUnsupported`
  * Missing Secret returns "not found" error (matches existing `errors.Is(err, fs.ErrNotExist)` or wrapped k8s NotFound — pick one and document)
  * Malformed `config` JSON returns parse error
  * `ListClusters` returns only cluster-typed Secrets
  * `HealthCheck` succeeds when ns accessible, fails when not
- `go build/vet/test ./...` clean
- New file fits existing `internal/providers/` style — review against `k8s_secrets.go` for naming and shape

**Effort:** M
