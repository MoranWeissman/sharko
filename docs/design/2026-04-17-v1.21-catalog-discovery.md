## 1. Problem

Today, Sharko's "catalog" has two meanings that are tangled together:

1. **The Git catalog file** — `configuration/addons-catalog.yaml` in the user's addons repo. It is the source of truth for what addons ArgoCD should render into ApplicationSets. Edited by hand or via `POST /api/v1/addons`.
2. **The Addons Catalog page in the UI** (`ui/src/views/AddonCatalog.tsx`) — browses the same file rendered with deployment-health stats. It is a dashboard, not a storefront.

Neither of these is a **discovery surface**. If an operator wants to add Prometheus to their platform, today they have to:

- Know the chart name (`kube-prometheus-stack`)
- Know the Helm repo URL (`https://prometheus-community.github.io/helm-charts`)
- Know a reasonable starting version
- Type all of that into the "Add Addon" modal

There is no browse, no search, no "what addons do other Sharko users run," no "what's the newest compatible version," no inline preview of values, no validation that the chart actually exists before the PR is opened. This is a raw-YAML experience dressed up as a form.

**v1.21 closes this gap.** Repurpose "Catalog" into the **browse-and-add** layer, in the Rancher Marketplace / OperatorHub sense, while keeping the current "deployed addons" view as a related-but-distinct tab ("Installed"). The existing Git-YAML file remains the source of truth; v1.21 adds discovery on top without changing the mutation model.

The goal is: **an operator can go from "I need cert-manager" to "a PR is open adding cert-manager" without leaving the Sharko UI and without knowing the chart URL in advance.**

| **Cyclops** | Internal catalog of `Module` templates (Helm charts or raw manifests) rendered as forms from `values.schema.json`. | UI browse of modules → dynamic form → deploy. | Add a template to Cyclops's module catalog (Git). |
| **Kyverno policy library** | Per-policy folder + `artifacthub-pkg.yml` metadata file; sha256 digest gates ArtifactHub pickup. ([kyverno/policies](https://github.com/kyverno/policies)) | Browse at [kyverno.io/policies](https://kyverno.io/policies/) and on ArtifactHub. | PR to `kyverno/policies`; admin renders markdown in a follow-up PR. Standard DCO. |
| **Sharko (today)** | Single `configuration/addons-catalog.yaml` in the user's addons repo. Flat list under `applicationsets:`. | UI page shows what's **deployed** (health, drift); no discovery, no search over a source of addon definitions. "Add Addon" is a blank form that requires the operator to already know the chart coordinates. | No public catalog. Each Sharko install has its own catalog file. |

**Key insights for Sharko:**

1. **No peer ships "a blank form that demands you type a chart URL."** Every serious product has a browsed, structured catalog first.
2. **The aggregator model (ArtifactHub) is already a solved problem.** Reinventing it would be wasteful. Sharko should **borrow** from it for the "newly-discoverable" case and **ship** a curated short-list for the "vetted defaults" case.
3. **Contribution paths vary:** Rancher / OperatorHub / Kyverno use centralized repos; Backstage distributes ownership; ArtifactHub just aggregates. Sharko's curated list should be centralized (CODEOWNERS-gated), and user-added addons remain in the user's own Git catalog.
4. **File-based (YAML / JSON, Git-friendly) is the dominant format.** CRDs are used only when cluster-level identity matters (OLM `CatalogSource`). A bundled YAML shipped in the image is consistent with CNCF norms and with Moran's steer.

## 4. Proposed design

### 4.1 Q1 — Catalog source format

**Recommendation: Hybrid. A curated YAML catalog shipped embedded in the Sharko container image is the default. An optional Git override can replace it. Entries are YAML, one file per addon, validated against a JSON Schema at build time.**

#### 4.1.1 File layout

```
internal/catalog/
├── catalog.yaml                 # index: ordered list of addons with metadata + paths
├── schema/
│   ├── addon.schema.json        # JSON Schema for a single addon entry
│   └── catalog.schema.json      # JSON Schema for the index
└── entries/
    ├── cert-manager.yaml
    ├── external-secrets.yaml
    ├── kyverno.yaml
    ├── kube-prometheus-stack.yaml
    ├── metrics-server.yaml
    ├── ingress-nginx.yaml
    └── ...
```

The `entries/` YAML files are **embedded** into the Sharko Go binary via `//go:embed`. `catalog.yaml` acts as an ordered, metadata-rich index. This keeps the bundle self-contained, reproducible per-release, and trivially verifiable from a signed container image.

#### 4.1.2 Schema (proposed)

Each `entries/<name>.yaml`:

```yaml
apiVersion: sharko.io/v1alpha1
kind: CatalogAddon
metadata:
  name: cert-manager
  displayName: "cert-manager"
  categories: [security, certificates]
  maintainers:
    - name: "The cert-manager Authors"
      url: "https://github.com/cert-manager/cert-manager"
  description: |
    Automatically provision and manage TLS certificates in Kubernetes.
  iconURL: "https://artifacthub.io/image/..."   # optional
  homeURL:  "https://cert-manager.io"
  docsURL:  "https://cert-manager.io/docs/"
  license: "Apache-2.0"
  deprecated: false
  superseded_by: ""                              # optional
spec:
  chart:
    repoURL: https://charts.jetstack.io
    name: cert-manager
    # versionPolicy: one of [pinned, latest-minor, any-release]
    # pinned = catalog authors pin a known-good version
    # latest-minor = resolve at runtime from ArtifactHub (cached)
    # any-release = let the user pick (default fallback)
    versionPolicy: latest-minor
    pinnedVersion: "1.16.3"       # used when versionPolicy=pinned or as fallback
  namespace: cert-manager
  syncWave: -1                    # opinionated default; user can override at add time
  defaults:
    # A small, opinionated values.yaml snippet recommended by the catalog authors.
    # The UI fills this into the editor as a starting point.
    installCRDs: true
    global:
      leaderElection:
        namespace: cert-manager
  dependsOn: []
  # Optional risk/compatibility matrix
  minKubernetesVersion: "1.23"
  artifacthubPackageID: "a9f7..."  # for one-click jump to ArtifactHub
signature:
  # RESERVED for v1.22+ — see §6 and §10
  cosign: ""
```

The `metadata` / `spec` / `signature` separation matches OLM FBC's plaintext style and makes the format trivially extensible without breaking v1.x consumers (unknown fields are ignored by the Go parser).

#### 4.1.3 Why not a CRD?

A `CatalogAddon` CRD was considered and rejected for v1.21:

- **It adds cluster-level identity we don't need.** Sharko's mutation model is PR-based; a CRD would shift the catalog to K8s-resident, which breaks the "everything flows through Git" story.
- **RBAC becomes harder, not easier.** We'd need to scope who can install which `CatalogAddon`, duplicating the v1.20 tier classification.
- **It couples Sharko to an API server for something that's just "a list."** Plain embedded YAML works behind air-gap with no moving parts.

A CRD may be reconsidered in **V3+** if Sharko gains an operator mode (see [V3 backlog](../../../.claude/projects/-Users-weissmmo-projects-github-moran-sharko/memory/project_v3_backlog.md)).

#### 4.1.4 Storage: image-bundled with optional Git override

- **Default:** The catalog ships embedded in the binary. Upgrading Sharko picks up new catalog entries. No network required at startup.
- **Override:** An admin can point Sharko at a Git repo via Helm value `catalog.source: git` + `catalog.git.repoURL` + `catalog.git.ref`. On startup, Sharko clones once into an emptyDir and watches for the configured ref. Same schema, same validation.
- **Both simultaneously:** `catalog.source: merged` merges bundled + Git, with Git winning on name collisions. Useful when an org wants to ship their own defaults without forking Sharko.

This mirrors Rancher's flexibility (built-in + external repos) without forcing an air-gapped user to depend on remote Git.

#### 4.1.5 Community contributions

- The bundled catalog lives at `internal/catalog/entries/` in the Sharko repo.
- Contributions are standard PRs with a `catalog/` label. A `CODEOWNERS` entry requires one maintainer review.
- Automated validation (CI): JSON Schema validation + each entry must successfully resolve against ArtifactHub (proves the chart actually exists).
- Deprecation is a YAML field (`deprecated: true`, optional `superseded_by:`), not a file deletion. Deprecated entries render greyed-out in the UI for one minor release, then are removed.
- A separate `sharko-catalog-extras` repo is **deferred** — not needed for v1.21. We can split later if volume justifies it (Rancher-style), but a single `entries/` tree is sufficient while the curated set is ~20-40 items.

### 4.2 Q2 — ArtifactHub integration

**Recommendation: Server-side proxy. Sharko's backend exposes `GET /api/v1/catalog/search?q=…` and `GET /api/v1/catalog/packages/{pkgID}`, which call ArtifactHub under the hood with caching. No direct browser → ArtifactHub calls.**

#### 4.2.1 Why server-side

| Concern | Client-side (browser → ArtifactHub) | Server-side (Sharko proxies) |
|---|---|---|
| Air-gap | Fails hard. No graceful degradation. | Disable cleanly, fall back to bundled catalog. |
| Corporate proxy | Depends on browser's proxy config + certificate store. | Single Sharko egress path — easier to configure once. |
| CORS | Reported to fail on some ArtifactHub endpoints ([community report](https://github.com/FairwindsOps/nova/issues/214)). | N/A — Sharko calls the API server-to-server. |
| Rate limit | Each operator's browser hits ArtifactHub independently. N users × ad-hoc traffic. | One Sharko instance + cache → one predictable source. |
| Cache | Per-browser only. | Shared across all UI/CLI users of one Sharko instance. |
| Supply chain story | Browser makes outbound calls to a third party — auditors dislike this. | All outbound network is from Sharko (already a reviewed dependency). |

ArtifactHub's own FAQ documents that "exact [rate limit] numbers are not documented because they are updated every now and then" ([ArtifactHub FAQ](https://artifacthub.io/docs/topics/faq/)) — we should assume the limits can tighten unpredictably. The safest pattern is exactly one server cache.

#### 4.2.2 Caching

Three-tier in-memory cache (Sharko is a single replica in v1.x; Redis is a V3 concern — see the [V3 backlog](../../../.claude/projects/-Users-weissmmo-projects-github-moran-sharko/memory/project_v3_backlog.md)):

| Layer | Key | TTL | Size cap |
|---|---|---|---|
| Search results | normalized query string | 10 min | 200 entries, LRU |
| Package summary | `pkgID` or `repo/chart` | 1 hour | 500 entries, LRU |
| Available versions | `pkgID` | 15 min (versions move) | 500 entries, LRU |

On cache miss, if the upstream call fails, we **serve stale** up to 24 h old with a `X-Cache-Stale: true` header. The UI shows a "(cached)" badge.

#### 4.2.3 Rate limit handling

Sharko backs off exponentially on `429`; the UI shows a non-modal toast ("ArtifactHub rate-limited — retrying in 30s"). We do not retry forever; after 3 attempts we surface the error and keep serving stale.

#### 4.2.4 Authentication

ArtifactHub's read-only endpoints don't require auth for search/package metadata. We do not ship an API token with Sharko. If ArtifactHub ever gates search behind auth, operators can configure `catalog.artifactHubToken` via the Connections UI — out of scope for v1.21.

#### 4.2.5 Search UX

- The search bar combines **curated catalog** (in-memory) and **ArtifactHub** (proxied).
- Results from the curated catalog render with a **"Curated" badge** + the catalog's opinionated default values.
- Results from ArtifactHub render with a **"ArtifactHub" badge** + the publisher's verified-status, star-count, and security-scan state (already exposed by ArtifactHub's API).
- Full-text search (name + description + keywords) is the default. A facet lets users filter to curated-only (which is what air-gapped installs always show).

### 4.3 Q3 — AI-assisted values annotation

**Recommendation: Ship it behind a feature flag, gated on existing Sharko AI config, cached per `(repo, chart, version)`, with a regex-based secret-leak guard before any upstream call.**

#### 4.3.1 What it does

When the user opens the values editor for an addon (or during the "Add addon" wizard), a button "Explain these values" calls `POST /api/v1/catalog/annotate` with the (chart, version). The server:

1. Fetches `values.yaml` for that chart version via `internal/helm/fetcher.go` (already implemented).
2. Runs a secret-leak pre-check (see §4.3.3).
3. Checks the cache for `(repo, chart, version)` — if hit, returns the cached annotated YAML.
4. If miss and AI is configured, calls the configured LLM with a deterministic prompt to insert inline `#` comments explaining each field. The prompt caps output size.
5. Stores the result in the cache.
6. Returns `{ annotated_yaml, tokens_used, cache: "hit"|"miss", model: "…" }`.

The UI shows the annotated output side-by-side with the editor. The user can choose to commit the annotated version or the bare version.

#### 4.3.2 Gating

- **If `ai.provider: none`** or the AI config isn't set: the button is hidden entirely (don't tease a feature the user can't use).
- **If AI is set up but the user lacks role `admin`**: show the button but it's read-only (no write to Git).
- **If air-gapped**: still works if Sharko is configured to hit an in-cluster Ollama (existing `ai.ollama.deploy` flag) or a local OpenAI-compatible endpoint (LiteLLM / vLLM). Otherwise, the button is disabled with an explainer.

#### 4.3.3 Secret-leak guard

Before any upstream LLM call, the raw `values.yaml` is scanned for patterns that *look* like secrets:

- 40+ char base64 strings (possible encoded creds)
- Strings matching `(?i)(api[-_ ]?key|token|secret|password|credential)[:= ]["']?[^\s"']{12,}` with a non-trivial value
- Strings matching common token prefixes: `sk-`, `ghp_`, `gho_`, `xoxb-`, `AKIA`, `AIza`, `-----BEGIN .* PRIVATE KEY-----`

If any match, the UI shows a **block dialog** explaining what was detected and lets the user either (a) remove/redact and retry, or (b) cancel. There is **no "send anyway" override** for v1.21 — the risk of leaking a PAT to a third-party LLM is larger than the benefit of inline comments.

#### 4.3.4 Caching

- **Cache backend:** A small BoltDB file (existing dependency candidates) or, preferably, a K8s `ConfigMap` named `sharko-ai-annotation-cache` (air-gap-friendly, no extra dep, survives pod restart). The key is `sha256(repo|chart|version)`, the value is the annotated YAML. ConfigMaps have a 1 MiB limit per object — we cap individual entries at 32 KiB and shard across N ConfigMaps if we exceed 200 entries. Given typical values.yaml sizes (5-20 KiB), this is plenty for a curated + common ArtifactHub set.
- **Invalidation:** Never. `values.yaml` for `cert-manager@1.16.3` is immutable. New versions get new cache entries.
- **Why not per-user:** The annotated output is a function of the inputs, not the caller. Sharing across users is correct and lowers token spend.

#### 4.3.5 Cost controls

| Control | Limit |
|---|---|
| Input size cap | 32 KiB raw `values.yaml`. Anything larger returns 413 + "too large to annotate". |
| Output size cap | `max_tokens` in the provider call (2000 for cloud, 3000 for Ollama). |
| Per-user rate limit | 10 annotations / hour (middleware, already in place for AI agent). |
| Global budget | Optional `ai.annotation.dailyBudgetUSD` — when set, Sharko refuses to call cloud providers after the cap. Ollama calls are always free. |

#### 4.3.6 Observability

Every annotation emits:

- A structured log line: `ai_annotate{provider, model, tokens_in, tokens_out, latency_ms, cache_hit}`.
- A Prometheus counter: `sharko_ai_annotations_total{cache_hit="true|false", provider="…"}`.
- A histogram: `sharko_ai_annotation_latency_seconds{provider="…"}`.

### 4.4 Q4 — Air-gap UX

**Recommendation: Hybrid — explicit `catalog.airGap: true` Helm value wins; auto-detect (3-second connect probe on startup) is the fallback when unset. The curated catalog is fully usable offline. Chart tarball fetches are documented to go through the operator's existing mirrored Helm registry.**

#### 4.4.1 Detection

1. **On startup**, if `catalog.airGap` is unset (`null`), Sharko dials `artifacthub.io:443` with a 3-second timeout. Unreachable → log a notice ("ArtifactHub unreachable — running in air-gap mode; set `catalog.airGap: true` to silence this probe") and set `airGap=true` in the runtime config.
2. **If `catalog.airGap: true` is set explicitly**, no probe runs. No outbound ArtifactHub calls are ever attempted.
3. **If `catalog.airGap: false` is set explicitly**, Sharko always tries and logs errors at WARN when the API is down.
4. A `POST /api/v1/catalog/reprobe` endpoint lets admins re-check without a pod restart. The UI has a "Retry connectivity" button in the catalog header.

#### 4.4.2 UI when offline

- ArtifactHub search input is **visible but disabled**, with a tooltip: "Disabled — ArtifactHub unreachable. [Retry connectivity]". (Not hidden, so the operator knows the feature exists.)
- Curated catalog browse works unchanged.
- AI annotation is disabled *unless* an in-cluster / on-prem LLM (Ollama, OpenAI-compatible) is configured.
- `values.yaml` preview on Add flow works if the chart tarball is reachable; otherwise shows "values preview unavailable — chart registry unreachable; defaults still applied."

#### 4.4.3 Chart tarball fetches in air-gap

This is the most subtle part. Sharko (today) fetches `index.yaml` and the chart tarball directly from `repoURL` during upgrade/values operations. In an air-gapped install, the operator has typically mirrored chart registries internally (Harbor, JFrog, Nexus, GitHub Enterprise, etc.). The catalog entry's `repoURL` in the user's `addons-catalog.yaml` points at the *internal* URL; the curated catalog entry points at the *upstream* URL.

For v1.21 we add a new Helm value `catalog.repoRewrites` (list of `{from, to}` pairs) that rewrites upstream `repoURL`s at catalog-render time. So an operator sets:

```yaml
catalog:
  airGap: true
  repoRewrites:
    - from: "https://charts.jetstack.io"
      to:   "https://harbor.internal.example/chartrepo/jetstack"
    - from: "https://charts.external-secrets.io"
      to:   "https://harbor.internal.example/chartrepo/external-secrets"
```

When the catalog is rendered to the user, `repoURL` shows the rewritten value. When "Add Addon" opens a PR, the PR writes the rewritten URL. This keeps the curated catalog portable while letting each org plug in its own mirror.

#### 4.4.4 Full air-gapped operator experience (end-to-end)

1. Operator installs Sharko from an internal registry with `catalog.airGap: true` + `catalog.repoRewrites: […]` + `ai.provider: ollama` (optional) + `ai.ollama.deploy: true`.
2. On startup, Sharko logs `catalog: airGap=true`; does not probe ArtifactHub.
3. UI opens. "Catalog" tab shows the bundled curated list, repo URLs already rewritten.
4. Operator clicks a tile → sees opinionated defaults + rewritten repo URL + pinned version → clicks "Add to fleet" → PR is opened against the org's Git addons repo via the existing v1.20 tiered-Git flow.
5. ArgoCD (in-cluster, also air-gapped) picks up the PR after merge and installs from the mirrored chart registry.
6. If the operator wants to annotate values with AI: Sharko calls its in-cluster Ollama sidecar with the chart's values.yaml; no external traffic.

No step in this flow requires internet access. This is the story we document in `docs/operator-manual/air-gap.md`.

### 4.5 Q5 — Curated catalog maintenance & contribution

**Recommendation: Ship bundled-with-release as the default. Open a single `sharko` repo tree (`internal/catalog/entries/`) for contributions. CODEOWNERS-gated review; CI validates schema + resolvability. Catalog signing deferred to v1.22+ (needs key management).**

#### 4.5.1 Ownership & review

- `.github/CODEOWNERS` entry: `internal/catalog/ @MoranWeissman` (solo for now; widen as trusted contributors join).
- Every PR to `internal/catalog/entries/` runs the `catalog-validate` CI job: JSON Schema validation, resolvability against ArtifactHub (proves `repoURL + chart` resolves; proves `pinnedVersion` exists), duplicate-name detection, dependency-cycle detection.
- A curator checklist (in `CONTRIBUTING-CATALOG.md`): license allow-list (Apache-2.0, BSD-3-Clause, MIT, MPL-2.0 — anything else flags for human review), security-scan status on ArtifactHub, maintainer activity recency.

#### 4.5.2 Versioning of entries

- Catalog entries use `versionPolicy: pinned | latest-minor | any-release`.
  - `pinned` — catalog authors pick a version; UI shows *only* that version.
  - `latest-minor` — catalog declares a major.minor floor (`1.16.*`); UI resolves the latest patch from ArtifactHub at display time.
  - `any-release` — UI lets the user pick from the full version list (this is the current behavior, preserved as a fallback).
- Pinned is the **default** for v1.21. Upgrades to pinned versions are PRs to the Sharko repo (easy to review, easy to roll back).

#### 4.5.3 Deprecation lifecycle

1. Mark `deprecated: true` + `superseded_by: "new-addon-name"` in the entry.
2. Next Sharko minor release: UI shows greyed-out tile + banner "This addon is deprecated. Use `new-addon-name`."
3. Following minor release: remove the file entirely.

This lets users who already deployed the old addon continue to see it in *their* `addons-catalog.yaml` — deprecation only affects *discovery*, not existing installs.

#### 4.5.4 Signing (deferred)

Catalog signing with cosign is genuinely appealing for CNCF alignment, but honestly — **we don't have a key-management story yet**. For v1.21 we:

1. Rely on the fact that the catalog is embedded in the signed Sharko container image (existing supply chain for the image protects the catalog).
2. Generate an `SBOM` entry that includes `internal/catalog/entries/` — the bundled catalog is already covered by the image's SBOM and cosign attestation.
3. Reserve a `signature.cosign` field in the schema for future per-entry signatures.

Per-entry signing (so a third-party Git-backed catalog override can be trust-verified) is a **V2 hardening** item, slotted next to external-vault for Git tokens. Both require proper KMS integration.

## 6. CNCF alignment

One of the purposes of v1.21 is to take Sharko from ~40% to ~50% on the [CNCF incubation maturity rubric](https://tag-security.cncf.io/community/assessments/guide/self-assessment/). The features below are designed to move specific line items.

### 6.1 Security & supply chain

- **SBOM coverage:** Bundled catalog is embedded in the Sharko container image; the existing image SBOM (CycloneDX) generated by our release workflow includes `internal/catalog/entries/*.yaml` contents by file hash. No new process required.
- **Cosign attestation:** Existing image cosign signature already covers the catalog. We add the catalog hash to the release attestation JSON.
- **No plaintext secrets in catalog entries:** schema forbids any field named `token`, `apikey`, `password`, `secret`, `credential` in `spec.defaults` — CI rejects PRs that try to add them.
- **Outbound traffic is audit-worthy:** every ArtifactHub and LLM call is logged (`internal/advisories/artifacthub.go` pattern already does this). Air-gap mode disables them completely and proves zero egress.
- **RBAC alignment:** discovery is readable by the `Viewer` role; adding from the catalog is a Tier 2 action (v1.20 classification) and requires `Operator`+.

### 6.2 Governance

- `CODEOWNERS` for `internal/catalog/` sets the contribution bar openly.
- A `CONTRIBUTING-CATALOG.md` specifies DCO, schema, license allow-list, deprecation policy. Matches Kyverno's model ([kyverno/policies CONTRIBUTING](https://github.com/kyverno/policies/blob/main/CONTRIBUTING.md)).
- Catalog PRs are merged only after CI green + one maintainer approval.
- CNCF CODE_OF_CONDUCT applies (we already reference it).

### 6.3 Observability

- **Metrics:** `sharko_catalog_requests_total{source=curated|artifacthub}`, `sharko_catalog_cache_hits_total`, `sharko_catalog_artifacthub_errors_total{status=…}`, `sharko_ai_annotations_total`, `sharko_ai_annotation_latency_seconds`. All scraped via `/metrics`.
- **Logs:** structured JSON via `log/slog`; one line per ArtifactHub call with correlation ID.
- **Traces:** out of scope for v1.21 (Sharko has no tracing integration yet — V2 hardening item).
- **Audit log (v1.18):** "added from catalog" emits with `catalog_entry_name`, `catalog_version_policy`, `chart_version` fields.

### 6.4 Accessibility (WCAG AA)

Target **WCAG 2.1 AA** for the new catalog UI:

- Keyboard navigation through tiles (tab/shift-tab, enter to open detail).
- ARIA labels on search bar, filter chips, version picker, "Add to fleet" button.
- Color contrast 4.5:1 for text, 3:1 for UI components — the existing shadcn/ui theme already meets this; verified via axe DevTools in CI.
- Focus-visible outlines on all interactive elements (already default in shadcn).
- Live-region announcement when the ArtifactHub search updates results.
- "Skip to results" link for screen-reader users.

This is the first Sharko view to explicitly target WCAG AA. Backfilling the rest of the UI is a **v1.22** item. Getting it right on a new view first is cheaper than retrofitting later.

### 6.5 API stability

- All new endpoints are under `/api/v1/catalog/*`. Swagger annotations on every handler (mandatory by project rule — `CLAUDE.md`).
- Response shapes use `json` tags; new fields are additive. Removals are a breaking change and require `v1beta` signaling + a deprecation window of one minor release.
- We add a top-level `apiVersion: sharko.io/v1alpha1` stamp on `CatalogAddon` entries so we have a principled path to `v1beta1` / `v1` as the schema stabilizes.

### 6.6 Documentation

Three audience-separated docs (per [docs-writer agent's ArgoCD-style target](../../../.claude/team/docs-writer.md)):

- `docs/user-guide/catalog.md` — how to browse and add addons.
- `docs/operator-manual/air-gap.md` — full air-gap setup including `repoRewrites`.
- `docs/developer-guide/contributing-catalog.md` — how to add / maintain a catalog entry.
- `CONTRIBUTING-CATALOG.md` at the repo root — quick-start for external contributors.

### 6.7 Vulnerability disclosure

Existing `SECURITY.md` covers Sharko itself. We add a section to `docs/operator-manual/air-gap.md` explaining that ArtifactHub is an untrusted remote source and pointing at [ArtifactHub's security-scan status](https://artifacthub.io/docs/topics/security/) as a signal — not a guarantee — of chart safety. Operators should treat any addon (curated or remote) as untrusted until they review it.

## 7. Out of scope for v1.21 (deferred to v1.22 / V2)

- **Per-entry cosign signatures** — requires key management we don't have. V2 hardening.
- **Catalog "rating" or "usage count"** — no telemetry on a pre-prod product; would demand privacy design we haven't done.
- **Third-party / community-driven separate catalog repo** — defer until curated set outgrows a single tree (>50 entries).
- **Catalog entries for non-Helm addons** (raw manifests, Kustomize) — v1.21 is Helm only.
- **Bundle / composition** (one-click "install prometheus + grafana + loki as a stack") — v1.22 candidate.
- **"Upgrade planner" / blast-radius UI** for catalog-driven version bumps — V3 backlog.
- **Webhook from ArtifactHub on new chart versions** — V3.
- **Direct install from the catalog** (no PR, no Git) — not considered. Breaks Sharko's mutation model.
- **Fine-grained RBAC on "who can see which catalog entries"** — V2.x scoped RBAC.

## 9. Open risks & unknowns

| # | Risk | Impact | Mitigation |
|---|---|---|---|
| R1 | **ArtifactHub rate limits are undocumented and can change without notice** ([ArtifactHub FAQ](https://artifacthub.io/docs/topics/faq/)). A popular Sharko deployment could hit them in the wild. | Search flakes, stale cache served. | Shared server cache (not per-browser), graceful stale-serve with UI badge, exponential backoff, optional API token via Connections. |
| R2 | **CORS behavior on ArtifactHub's API is not uniformly documented and has been reported to fail on some endpoints.** | Irrelevant — we're server-side — but this is the reason we chose server-side. | Decision recorded in §4.2.1. |
| R3 | **Secret-leak regex in AI annotate can false-positive (block a legit `password:` comment) or false-negative (miss a novel token format).** | False-positive = friction. False-negative = data leak. | Err hard toward false-positive. No "send anyway" for v1.21. Treat any pattern match as a block. |
| R4 | **ConfigMap cache for AI annotations has a 1 MiB object cap.** | Cache fills, oldest-wins doesn't work cleanly in ConfigMap without custom logic. | Shard across N ConfigMaps (`sharko-ai-annotation-cache-0..N`), 200 entries each. Alternative: BoltDB file on a PVC if we already have one. |
| R5 | **`repoRewrites` can silently mis-route a chart.** If an operator typos an `to:` URL, Sharko would emit PRs pointing at a non-existent mirror, and the operator might not notice until ArgoCD can't sync. | Silent broken installs. | Validate rewrites on startup (`HEAD` request against each `to:` URL); fail startup if unreachable **unless** `airGap=true`. In air-gap mode, log + continue. |
| R6 | **Catalog entries lacking `pinnedVersion` but set to `versionPolicy: latest-minor` depend on ArtifactHub resolving.** In air-gap mode, these entries have no resolvable version. | Air-gapped install sees "no version available". | Require `pinnedVersion` as a fallback on every entry, regardless of policy. Lint rule in `catalog-validate` CI. |
| R7 | **CNCF accessibility expectations are less formalized than security/supply-chain.** WCAG AA is aspirational; we're guessing at the bar. | Over- or under-investment. | Pick WCAG 2.1 AA as a concrete target. Document it as "our bar", not "CNCF's bar." Revisit after real contributor/user feedback. |
| R8 | **AI annotation token spend could surprise operators if defaults are too loose.** | Bill shock. | `ai.annotation.dailyBudgetUSD` default = `0` (unlimited but warn); document the rate limit; Ollama is free. |
| R9 | **"Curated catalog" drift vs. real-world addon versions.** If we pin `cert-manager@1.16.3` and the community moves to 1.18, our curated tile looks stale. | UX erosion. | Quarterly bump cadence (calendar reminder); `versionPolicy: latest-minor` for addons where authors explicitly opt in. |
| R10 | **We haven't verified whether every proposed curated addon accepts Apache-2.0-or-compatible licenses.** | License friction for a CNCF project. | `catalog-validate` CI enforces license allow-list. Exceptions require a README entry. |
| R11 | **ArtifactHub documents that exact rate limits are undocumented and vary.** Confirmed from the FAQ. No hard numbers to design against. | Design on assumptions. | Assume conservatively (~10 req/s, ~1000/hr per IP); measure once we have real traffic. |
