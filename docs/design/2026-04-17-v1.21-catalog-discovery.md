## Revision history

The 2026-04-17 draft framed several over-engineered options as live decisions. This revision locks the simplified design after the 2026-04-18 brainstorm. Major changes:

- **Catalog lives in the main Sharko repo**, not a separate `sharko-catalog` repo — single file `catalog/addons.yaml`, embedded via `//go:embed`.
- **Schema flattened** to a single Sharko-native struct per entry (no `apiVersion/kind/metadata/spec`). Rationale: the catalog is metadata-only, not a K8s resource; FBC-style envelope was overhead with no payoff.
- **Catalog is metadata/references only.** Sharko does not host, proxy, or mirror Helm charts — ArgoCD still pulls from upstream at deploy time. This means `catalog.airGap`, `catalog.repoRewrites`, and ConfigMap overrides for the Sharko-native catalog are removed; air-gap is the user's infra problem.
- **No database / BoltDB / ConfigMap cache for AI annotations.** Git is the cache: once an addon is annotated, the annotated file lives in the user's addons repo and is read on every subsequent render.
- **No per-entry cosign.** Release-level cosign on the Sharko binary covers the embedded catalog by construction.
- **Always-visible "Pull upstream defaults" button** (shipped in v1.20) is replaced by a contextual **version-mismatch banner** on the Values tab.
- **Marketplace UX finalized** as a 3-tab modal: Browse (default) / Search by name / Paste Helm URL.
- **Smart values layer is in scope for v1.21** (not deferred): heuristic cluster-specific-field detection + optional LLM fallback + global/template split.
- **OpenSSF Scorecard integration added.** Note: Scorecard publishes a **0-10 numeric score**, not letter grades (corrected from the brainstorm shorthand). Displayed in the UI as a badge with the numeric score plus a coarse tier label Sharko derives (e.g., "8.2 / Strong").
- **Cosign scope widened** to cover the release pipeline itself: image + binary + release artifacts are cosign-keyless signed; catalog integrity follows.

Items considered and rejected live in §6.

## 1. Problem

Sharko v1.20 added a values editor, tiered attribution, per-user PAT. What it did not add is any way to **discover** addons. An operator who wants cert-manager on their fleet today has to:

- Know the chart name (`cert-manager`)
- Know the Helm repo (`https://charts.jetstack.io`)
- Know a reasonable starting version
- Type those into the "Add Addon" modal, which is a blank form with no validation that the chart exists

The existing Addons Catalog page (`ui/src/views/AddonCatalog.tsx`) is a **dashboard** over what is already deployed — it is not a storefront. Nothing in Sharko tells a first-time user which addons are common, which are CNCF-graduated, or what `values.yaml` they'll inherit.

**v1.21 fixes this.** The Addons Catalog page grows a Marketplace entry point: one "Add Addon" button opens a modal with three paths — browse a curated list, search by name, or paste a Helm URL — all of which end in the same "open a PR to `addons-catalog.yaml`" flow that v1.20 shipped. No new mutation model; a new discovery surface.

Goal: **"I need cert-manager" to "a PR is open adding cert-manager" in under a minute, without knowing the chart URL in advance.**

Alongside discovery, v1.21 ships a **smart values layer** that generates a sensible `addons-global-values/<addon>.yaml` at add-time and refreshes it when the chart version moves, with an AI-annotate option for inline field documentation.

**Insights that drove v1.21's shape:**

1. **No peer ships "a blank form that demands a chart URL."** Every serious product has a browsed, structured catalog first.
2. **Aggregator (ArtifactHub) is already a solved problem.** Sharko borrows from it for the "newly-discoverable" case and ships a curated short-list for the "vetted default" case.
3. **Centralized contribution (Rancher, OperatorHub, Kyverno) beats distributed (Backstage) for curation quality, but loses on throughput.** v1.21 picks centralized because our curated set is ~50 entries, not thousands.
4. **Plain YAML wins** over CRDs, FBC envelopes, or xpkg archives for something that is just "a list." YAML ships in the image, works offline, round-trips cleanly in Git.

## 4. Locked design

### 4.1 Catalog format

**Single file: `catalog/addons.yaml` in the main Sharko repo**, embedded into the Go binary via `//go:embed` at build time. Metadata/references only — no chart tarballs, no Helm index mirroring, no external hosting.

```
sharko/
├── catalog/
│   ├── addons.yaml          # ~50-100 curated entries
│   └── schema.json          # JSON Schema for validation
├── internal/
│   └── catalog/
│       ├── embed.go         # //go:embed catalog/addons.yaml
│       ├── loader.go        # parse + validate + index
│       ├── search.go        # in-memory search
│       └── scorecard.go     # OpenSSF Scorecard refresh
```

**Why one file, not one-file-per-entry:**

- Easier to diff and review in a PR.
- Easier to scan for duplicates / ordering during review.
- At 50-100 entries with ~15 fields each, a single YAML is ~1500-2000 lines — comfortable for editor and Git.
- We can split later if volume justifies it; YAML + JSON Schema ports 1:1 to a directory tree.

**Why embedded, not a separate repo or image:**

- Updates ship with Sharko releases. Cadence is fine — we're not a CVE database.
- No runtime network dependency, no separate signing story.
- Offline by construction.
- Release-level cosign on the Sharko image + binary already covers catalog integrity (see §5.7).

**Updating the catalog** is a PR to `catalog/addons.yaml` in the main repo, CODEOWNERS-gated to the maintainer. The catalog does not ship independently of Sharko releases.

### 4.2 Schema (Sharko-native, flat)

Each entry:

```yaml
- name: cert-manager
  description: Automated TLS certificate lifecycle management for Kubernetes.
  chart: cert-manager
  repo: https://charts.jetstack.io
  default_namespace: cert-manager
  default_sync_wave: 0
  docs_url: https://cert-manager.io/docs
  homepage: https://cert-manager.io
  source_url: https://github.com/cert-manager/cert-manager
  maintainers: [jetstack]
  license: Apache-2.0
  category: security                # single primary category (see §4.2.1)
  curated_by:                       # trust signals — tag list
    - cncf-graduated
    - aws-eks-blueprints
    - artifacthub-verified
  security_score: 8.3               # OpenSSF Scorecard aggregate, 0-10; "unknown" if not queried
  security_score_updated: "2026-04-15"
  github_stars: 12500
  min_kubernetes_version: "1.23"
  deprecated: false
  superseded_by: ""                 # optional
```

Notes:

- `security_score` is **numeric 0-10** (corrected from letter grade); UI derives a tier label (see §5.6). Value `unknown` is allowed.
- `curated_by` entries are a closed enum: `cncf-graduated`, `cncf-incubating`, `cncf-sandbox`, `aws-eks-blueprints`, `azure-aks-addon`, `gke-marketplace`, `artifacthub-verified`, `artifacthub-official`. Extending the enum is a schema-PR with a public criterion.
- No `signature` field, no `apiVersion`, no `kind`. Plain Sharko-native struct.
- Unknown fields are **tolerated** in the loader (future-compatible additions to the schema do not break older Sharko binaries).

#### 4.2.1 Categories (closed enum)

`security`, `observability`, `networking`, `autoscaling`, `gitops`, `storage`, `database`, `backup`, `chaos`, `developer-tools`. New categories require a schema PR.

### 4.3 Marketplace UX flow

"Add Addon" on `AddonCatalog.tsx` opens a modal with **3 tabs**:

#### 4.3.1 Browse marketplace (DEFAULT tab)

- Grid of curated tiles from `catalog/addons.yaml`.
- Filters: **Category** (pill), **curated_by** (multi-select), **Security score** (min slider, 0-10), **Min K8s version** (select).
- Search box searches name + description + maintainers.
- Tile click → **Configure** step (§4.3.4).

#### 4.3.2 Search by name

- Search input; on type (debounced 250 ms), Sharko backend calls:
  1. Local curated index — name/description match
  2. ArtifactHub `/packages/search?ts_query_web=<q>&kind=0` (Helm) — proxied server-side
- Results merged: curated hits first (badge "Curated"), ArtifactHub hits second (badge "ArtifactHub" + verified-publisher/star-count from the API).
- Click a result → Sharko fetches `/packages/helm/{repo}/{chart}` for metadata → **Configure** step.
- **Air-gap / ArtifactHub unreachable:** the tab still shows curated-only matches with a banner: "ArtifactHub unreachable — showing curated only. Retry connectivity."

#### 4.3.3 Paste Helm URL (power-user)

- Two inputs: **Chart repo URL**, **Chart name**.
- On URL blur or name blur: Sharko fetches `<repoURL>/index.yaml`, parses it via `internal/helm/fetcher.go`, confirms the chart exists.
- Success: green check + "Found 12 versions (latest 1.20.2)". Failure: error + fix hint.
- On "Continue" → **Configure** step.

#### 4.3.4 Configure step (shared by all three paths)

Fields, prefilled from catalog entry / ArtifactHub metadata / `index.yaml`:

- **Addon name** (editable; default = chart name).
- **Namespace** (editable; default = `default_namespace` or chart name).
- **Sync wave** (editable; default = `default_sync_wave`).
- **Version** — picker shows top 5 stable versions from the chart's `index.yaml` (skip pre-releases unless user toggles "Show pre-releases"). User can also type a version; Sharko validates it resolves.
- **Cluster selector** — multi-select of clusters from the existing ApplicationSet model; default = all managed clusters.

**No diff preview** — consistent with the Values editor (v1.20). GitHub renders the PR diff.

**Duplicate guard:** if `name` already exists in the user's `addons-catalog.yaml`, submit is blocked: "**`cert-manager`** is already installed. Go to the addon page to manage it."

#### 4.3.5 Submit

- Sharko opens a PR that (a) adds the entry to `configuration/addons-catalog.yaml`, (b) creates `configuration/addons-global-values/<addon>.yaml` from the smart values layer (§5).
- Tier 2 attribution — reuses `GitProviderForTier` pattern (`internal/api/tiered_git.go`).
- Toast: "PR opened — [View on GitHub]".

### 4.4 Smart values layer

When an addon is added from the marketplace (or its chart version is bumped), Sharko regenerates the addon's values files from the upstream chart's `values.yaml`. This is **in scope for v1.21** (not deferred).

#### 4.4.1 Pipeline

1. **Fetch `values.yaml`** from the chart artifact. Plumbing already exists: `internal/helm/fetcher.go:188` (`FetchValues`).
2. **Smart parse — detect cluster-specific fields.** A field is flagged cluster-specific when it matches either the heuristic rules or the LLM (when AI is enabled).
3. **AI annotate** (when AI configured and not opted out): insert inline `#` comments explaining each field. This mirrors the pattern Moran maintains in the `argocd-addons-platform` reference repo.
4. **Split into two outputs**:
   - **Global values file** (`configuration/addons-global-values/<addon>.yaml`) — upstream defaults, annotated, with cluster-specific fields **removed** (left commented-out at the field's original position).
   - **Per-cluster template** — appended as a trailing commented block at the bottom of the same global values file. Format:
     ```yaml
     # --- per-cluster overrides template ---
     # Uncomment and customize in configuration/addons-clusters-values/<cluster>/<addon>.yaml
     # ingress:
     #   host: <cluster-specific>
     # resources:
     #   requests:
     #     cpu: "100m"
     #     memory: "128Mi"
     ```
5. **File header** on every generated values file:
   ```yaml
   # Generated by Sharko from cert-manager@v1.20.2 on 2026-04-19
   # Chart source: https://charts.jetstack.io
   # AI annotation: enabled
   # sharko: managed=true
   ```
6. **New-cluster seeding** — when a user later enables an existing addon on a new cluster, Sharko reads the commented template block from the global file and seeds `configuration/addons-clusters-values/<cluster>/<addon>.yaml` with the same uncommented lines (as a starting point; user edits + merges).

#### 4.4.2 Cluster-specific field heuristics

Fields are flagged cluster-specific when their **full YAML path** matches any of (case-insensitive, dotted-path glob):

- `*.host`, `*.hostname`, `*.domain`, `*.endpoint`, `*.url` (connection + ingress targets)
- `*.ingress.*`, `*.tls.*` (TLS / ingress config)
- `*.replicaCount`, `*.replicas`
- `*.resources.*` (requests / limits — commonly cluster-sized)
- `*.persistence.*`, `*.storageClass*`, `*.pvc.*`
- `*.nodeSelector`, `*.tolerations`, `*.affinity*`
- `*.externalSecret*`, `*.existingSecret*`, `*.secretName`
- `*.clusterName`, `*.region`, `*.environment`
- `*.serviceAccount.annotations.*` (IRSA / Workload Identity)

**This list is the starting point** — maintainer approval of the full list is **Decision #2** below. The list is in the Sharko config (`internal/catalog/heuristics.go`) and a schema-PR can extend it.

#### 4.4.3 LLM fallback

When AI is configured (any non-`none` provider) and the chart's version has not been processed before:

- Sharko sends `values.yaml` to the LLM with a deterministic prompt: "For each top-level YAML field below, answer 'cluster-specific: yes/no' with a one-line reason."
- Result is merged with the heuristic output (union — LLM additions are treated as additional cluster-specific fields).
- **One LLM call per chart version.** The resulting annotated YAML is the cache — it's committed to the user's Git repo. No ConfigMap cache, no BoltDB; Git is the cache.
- Secret-leak pre-scan on the raw `values.yaml` before any upstream LLM call (pattern list matches the v1.20 values-editor guard — `sk-`, `ghp_`, `AKIA`, `AIza`, 40+ char base64, PEM blocks, `(?i)(api.?key|token|secret|password|credential)[=:].{12,}`). Any match hard-blocks the call; UI tells the user what was detected.

**When AI is not configured:** heuristic-only. UI shows a banner on the Values tab: "AI annotation not configured — values are not commented. Configure AI in Settings → AI to enable."

#### 4.4.4 AI annotation opt-out

- **Global:** Settings → AI → "Annotate values on generate" toggle (default ON when AI is set up).
- **Per-addon opt-out:** first-time writers of the generated file can add `# sharko: ai-annotate=off` to the file header, which Sharko respects on future regenerations.
- **Forbidden on secrets:** `values.yaml` content detected as secret-bearing (pre-scan) never reaches the LLM regardless of toggle.

#### 4.4.5 Version-mismatch detection (replaces v1.20 always-visible "Pull upstream" button)

- Sharko parses the `# Generated by Sharko from <chart>@<version>` header on every render of `addons-global-values/<addon>.yaml`.
- If the `version` in the user's `addons-catalog.yaml` differs from the header's version, the **Values tab shows a banner**:
  > ⚠ Chart upgraded to v1.20.2 — values were generated for v1.19.0. Refresh values from upstream?
  >
  > [Refresh now] [Dismiss]
- Click "Refresh now" → same pipeline as §4.4.1 → opens a Tier 2 PR rewriting `addons-global-values/<addon>.yaml` with the new header + annotated defaults + template block.
- **Existing per-cluster overrides are never clobbered.** Only the global file + trailing template section are regenerated. Files in `addons-clusters-values/*/` are left alone.

#### 4.4.6 What was in v1.20 and is removed

- **Always-visible "Pull upstream defaults" button** on the Values tab (`ui/src/components/ValuesEditor.tsx`) — removed. Rationale: noisy when there's nothing to refresh; misleading when it would overwrite user edits. Replaced by the version-mismatch banner, which only appears when there's an actual mismatch.

### 4.5 ArtifactHub integration (backend proxy)

Sharko's backend proxies ArtifactHub — the browser never calls it directly.

**Endpoints (new, all swagger-annotated):**

- `GET /api/v1/catalog/search?q=<term>` — blended curated + ArtifactHub results.
- `GET /api/v1/catalog/entries/{name}` — single curated entry.
- `GET /api/v1/catalog/remote/{pkgID}` — ArtifactHub package detail.
- `GET /api/v1/catalog/versions?repo=<url>&chart=<name>` — `index.yaml`-derived version list.
- `POST /api/v1/catalog/reprobe` — force connectivity re-check.

**3-tier in-memory cache** (single replica, no Redis):

| Layer | Key | TTL | Cap |
|---|---|---|---|
| Search results | normalized query | 10 min | 200 entries (LRU) |
| Package summary | `pkgID` | 1 hour | 500 entries (LRU) |
| Available versions | `repo|chart` | 15 min | 500 entries (LRU) |

**Stale-serve on upstream failure** — stale up to 24 h with `X-Cache-Stale: true`; UI shows "(cached)" badge. **Exponential backoff on 429** (3 attempts; then stale-serve + toast).

**No API token** — ArtifactHub's read endpoints are public. If they ever gate search behind auth, users configure `catalog.artifactHubToken` via Connections (V2 concern).

### 4.6 OpenSSF Scorecard integration

- **Source:** `https://api.scorecard.dev/projects/github.com/<owner>/<repo>` — public API, aggregate score 0-10.
- **Refresh:** runtime, daily at 04:00 UTC, for every entry whose `source_url` points at github.com. In-memory cache, re-populated on pod restart from whatever the catalog ships with. Failures are non-fatal — the catalog's baked-in `security_score` serves until the next successful refresh.
- **UI display:** numeric score + coarse tier derived by Sharko:
  - `>= 8.0` → Strong (green)
  - `5.0 – 7.9` → Moderate (amber)
  - `< 5.0` → Weak (red)
  - `unknown` → grey
- **Filter:** Browse tab has a "Min score" slider (0-10, default 0).
- **Metrics:** `sharko_scorecard_refresh_total{status}`, `sharko_scorecard_last_refresh_timestamp`.

Scorecard's weekly public scan omits three checks at scale (CI-Tests, Contributors, Dependency-Update-Tool) — the aggregate is still meaningful. We document this caveat in the user guide.

### 4.7 Cosign (release-level only)

- The Sharko release pipeline (`.github/workflows/release.yml`) adds **cosign keyless signing** (GitHub OIDC identity) for:
  - The Docker image (`ghcr.io/moranweissman/sharko:<tag>`)
  - The Linux/macOS binaries attached to the GitHub release
  - The `catalog/addons.yaml` file as a release asset (optional but cheap)
- **Per-entry signing is explicitly rejected** (§6). Rationale: the catalog ships embedded in the signed binary; once the binary's signature verifies, the catalog's bytes are trusted by construction. Per-entry cosign adds a KMS story we don't need.
- Verification docs go in `docs/operator-manual/supply-chain.md`.

### 4.8 Accessibility (WCAG 2.1 AA — new pages only)

The marketplace modal + smart-values banner UI target **WCAG 2.1 AA**:

- Keyboard navigation through tiles (tab/shift-tab; enter opens Configure).
- ARIA labels on search bar, tabs, filter chips, version picker, Submit button.
- Color contrast 4.5:1 for text, 3:1 for UI components (shadcn/ui default satisfies this; verified with axe DevTools).
- Focus-visible outlines (shadcn default).
- Live-region announcement on search-result updates.
- Dialog + tablist roles set correctly per WAI-ARIA Authoring Practices.

**Existing pages (Dashboard, ClusterDetail, AddonDetail, Settings) are not retrofitted in v1.21.** Backlog item for v1.22.

### 4.9 Contribution model

- **Single repo.** `catalog/addons.yaml` + `catalog/schema.json` live in the main Sharko repo. No `sharko-catalog` satellite repo (§6).
- **CODEOWNERS** entry: `catalog/** @MoranWeissman` (solo for now; widen as trusted contributors emerge).
- **CI gate** — `.github/workflows/catalog-validate.yml`:
  - JSON Schema validation (structural).
  - Chart resolvability — for each entry, `GET <repo>/index.yaml` + confirm `<chart>` is listed.
  - Duplicate-name detection.
  - License allow-list (Apache-2.0, BSD-3-Clause, MIT, MPL-2.0). Anything else flags for human review in the PR comment.
- Deprecation is a YAML field (`deprecated: true`, optional `superseded_by:`), not file deletion. Deprecated entries render greyed-out in Browse for one minor release, then are removed.

## 6. What was considered and rejected

From the earlier drafts and the 2026-04-18 brainstorm, these options were explicitly ruled out:

| # | Considered | Rejected because |
|---|---|---|
| 1 | Separate `sharko-catalog` repo | Unnecessary fragmentation at 50-100 entries; the catalog is not released independently of Sharko. |
| 2 | CRD-based catalog (`CatalogAddon`) | Adds cluster identity we don't need; breaks the "everything flows through Git" story. May revisit in V3+ if Sharko gains an operator mode. |
| 3 | ConfigMap override for Sharko's native catalog | Dual storage with no use case — users edit `addons-catalog.yaml` in their own repo, which is already the override. |
| 4 | `catalog.airGap` Helm value | The catalog is metadata-only; Sharko never fetches chart tarballs on the user's behalf. Air-gap is the user's infra problem, handled by their internal Helm mirror. |
| 5 | `catalog.repoRewrites` Helm value | Same reason. If the user mirrors charts internally, they put the mirrored URL in their own `addons-catalog.yaml`. |
| 6 | Database / SQLite / BoltDB / Postgres for state | Sharko stays stateless. Git is the source of truth for user data; in-memory caches are fine for ephemeral state. |
| 7 | Per-entry cosign signatures on catalog entries | Release-level cosign on the Sharko image + binary already covers the embedded catalog. Per-entry needs a KMS story we don't want. |
| 8 | ConfigMap cache for AI annotations | Git is the cache — annotated values live in the user's addons repo. |
| 9 | Always-visible "Pull upstream defaults" button (shipped v1.20) | Noisy when there's nothing to refresh, confusing when it would overwrite user edits. Replaced by contextual version-mismatch banner. |
| 10 | Non-Helm addon support (raw manifests, Kustomize, OLM operators) | v1.21 is Helm-only. Crossplane / operator-based CNCF projects deferred. |

## 9. Out of scope (v1.22 / V2)

- **Third-party / private catalog repositories** — single curated catalog in main repo for v1.21.

- **Per-entry cosign signatures** — see §6.
- **WCAG AA retrofit on existing pages** — v1.22.

- **Telemetry-backed "usage count" / popularity ranking** from real Sharko installs — requires telemetry design.

- **Fine-grained RBAC on "which catalog entries a user can see"** — V2.x scoped RBAC.
- **Direct install from catalog (no PR)** — explicitly rejected; breaks Sharko's mutation model.

## 10. Open risks

| # | Risk | Impact | Mitigation |
|---|---|---|---|
| R1 | **ArtifactHub rate limits are undocumented** ([FAQ](https://artifacthub.io/docs/topics/faq/)) — a popular Sharko deployment could trip them. | Search flakes, stale-serve kicks in. | Shared server cache; stale-serve; exponential backoff; optional user-supplied token. |
| R2 | **OpenSSF Scorecard weekly scan omits 3 checks** at scale (CI-Tests, Contributors, Dependency-Update-Tool). Aggregate is slightly less accurate than a local run. | Score may under-reflect reality. | Document the caveat; let users click through to the full Scorecard UI for the detailed breakdown. |
| R3 | **Smart-parser heuristics false-positive** (flag a field as cluster-specific that should be global) or **false-negative** (miss a new chart pattern). | User sees an odd split; edits manually. | Heuristic list is version-controlled (`internal/catalog/heuristics.go`); LLM fallback catches novel patterns when AI is on; user can always edit the generated file and `# sharko: ai-annotate=off` locks it. |
| R4 | **LLM secret-leak guard false-positive** (block a legitimate `password` comment) or **false-negative** (miss a novel token format). | Friction or data leak. | Erring hard toward false-positive; no "send anyway" override; identical pattern list to v1.20 values editor. |
| R5 | **Curated catalog drift** — we pin `cert-manager@v1.20.2` in the tile; community moves to v1.22. | Tile looks stale. | Version picker always shows the real top-5 from `index.yaml`; catalog's default is a **suggestion**, not a pin. Quarterly catalog-refresh PR. |
| R6 | **License allow-list mis-flags** an Apache-2.0-licensed chart because SPDX strings vary (`Apache-2.0` vs `Apache License 2.0`). | CI noise. | Allow-list matches SPDX canonical IDs; README explains how to normalize. |
| R7 | **Embedded catalog grows unwieldy** if we approve every PR. | File review friction. | Soft cap at 100 entries; split into directory tree when we cross that. Curation bar documented in `CONTRIBUTING-CATALOG.md`. |
| R8 | **Scorecard API downtime** during the daily refresh. | `security_score` stays at last-known value for a day. | Non-fatal failure; last known value + `security_score_updated` timestamp. Metric emits. |
| R9 | **Version-mismatch banner misfires** if the user hand-edits the header in the generated file. | Banner stuck on or off. | `# sharko: managed=true` line in the header; if absent, banner is suppressed and a tooltip explains why. |
