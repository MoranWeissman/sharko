## 1. Motivation

v1.21 shipped a single embedded catalog (`catalog/addons.yaml`, `//go:embed`). v1.22 closed the polish gap — marketplace a11y, multi-arch image, audit-log retention, docs screenshots — but did not open the catalog.

The natural next ask from orgs with internal Helm charts or stricter trust models is: **extend the catalog without forking Sharko.** They want to list their own internal charts alongside the curated set, prove provenance on the vendor-shipped list, and keep the curated list fresh without the maintainer chasing every CNCF graduation event manually.

v1.23 opens three extensibility paths that compose cleanly with the existing curated catalog:

1. **Third-party private catalogs** (`SHARKO_CATALOG_URLS` override) — merge extra YAML catalogs under the embedded one.
2. **Per-entry cosign signing** — Sigstore-backed provenance on each catalog entry, not just the release binary.
3. **Automated trusted-source scanning bot** — nightly GitHub Action opens PRs to `catalog/addons.yaml` based on CNCF Landscape, EKS Blueprints, and other upstream lists.

None of the three is a breaking change. The embedded catalog remains the baseline; all three subsystems are opt-in. Subsystems A and B are independent at the feature level — A works without B (unsigned catalogs merge, every entry flagged `verified: false`) and B works without A (per-entry signatures on the embedded catalog). A uses B only when both are present and a sidecar or per-entry signature is configured; the coupling is a runtime delegation via the entry point described in §3.3.1, not a build-time dependency.

## 2. Subsystem A — Third-party private catalogs (`catalog.url` override)

### 2.1 Config surface

- **Env var:** `SHARKO_CATALOG_URLS` — comma-separated HTTPS URLs. Example: `SHARKO_CATALOG_URLS=https://catalogs.example.com/internal.yaml,https://catalogs.example.com/partner.yaml`.
- **Refresh cadence env var:** `SHARKO_CATALOG_REFRESH_INTERVAL` — Go duration, bounded to `[1m, 24h]`, default `1h`.
- **SSRF guard:** the loader validates every configured URL at startup — scheme must be `https`, and the resolved host must not fall into loopback, RFC1918 private, IPv6 ULA (`fc00::/7`), link-local (`169.254/16`, `fe80::/10`), or unspecified ranges. Classification uses `net/netip` over `net.LookupHost`. A deliberate opt-out env var `SHARKO_CATALOG_URLS_ALLOW_PRIVATE=true` is provided for home-lab / dev scenarios — it is unsafe on untrusted networks and Sharko logs a WARN when it is enabled. This guard folds in the v1.22 code-review concern that a privileged catalog puller could otherwise be aimed at internal services (cloud metadata endpoints, ArgoCD's own API, etc.).
- **Settings page:** **resolved — env-only for v1.23.** The Settings page surfaces a read-only view of the configured sources; editing is out of scope until v1.24. See §7.1.

The env var is the single source of truth; the loader produces one in-memory list of sources.

### 2.2 Fetch + merge strategy

On startup and every 1 h (configurable via `SHARKO_CATALOG_REFRESH_INTERVAL`):

1. For each URL, `GET` the YAML over HTTPS (respect `HTTP_PROXY` / `HTTPS_PROXY` / `NO_PROXY` env vars).
2. Validate against `catalog/schema.json` — same schema the embedded catalog uses (bumped to v1.1 per §5).
3. Merge under the embedded catalog with this conflict rule: **embedded wins for same `name`**, and the UI surfaces an "overridden by internal curation" note on the third-party tile.
4. If fetch fails, keep the last successful snapshot for that URL; surface status in `GET /api/v1/catalog/sources`.

### 2.3 Source attribution

Each entry in the in-memory index carries a `source` field:
- `embedded` — shipped in the Sharko binary
- `<URL>` — one of the configured third-party URLs

The Browse tab surfaces a source badge on each tile (visible in detail page too). Tiles from third-party sources also show a verification chip — `Verified by <issuer>`, `Unverified`, or `Source: embedded` — computed from Subsystem B.

### 2.4 Signature verification (ties to Subsystem B)

If a configured URL has a companion `.sig` / `.bundle` sidecar, Subsystem B's verifier is invoked on the fetched YAML bytes before merge. See §3.3 for the verification flow. Without Subsystem B configured, sidecar URLs are fetched but treated as unsigned (flagged `verified: false`). Subsystem A does not implement cosign itself — it delegates to Subsystem B's entry point when both are present and a sidecar is configured; otherwise it proceeds with an unsigned merge. See §3 for the full trust model.

### 2.5 Refresh cadence

- Startup: fetch all URLs; failures are non-fatal (last-known snapshot serves).
- Periodic: every 1 h default, configurable.
- Manual: a `POST /api/v1/catalog/sources/refresh` admin endpoint forces an immediate refresh of all sources (Tier 2, audited).

### 2.6 API surface

- `GET /api/v1/catalog/sources` — returns `[{url, status: ok|stale|failed, last_fetched, entry_count, verified}]` per source (embedded source is always included with `url: "embedded"`).
- `POST /api/v1/catalog/sources/refresh` — force refresh; audited.

### 2.7 Caching

In-memory only. Stateless principle (NFR-V121-1) is preserved — we do not persist third-party catalogs to disk, ConfigMap, or database. A restarted pod re-fetches on startup.

## 3. Subsystem B — Per-entry cosign signing

### 3.1 Motivation

Third-party catalog publishers need to prove provenance. Consumers need to pin trust to specific Sigstore identities (e.g., "only accept entries signed by `https://github.com/cncf/.*/.github/workflows/.*`").

Release-level cosign (shipped v1.21, §4.7) covered the embedded catalog because the catalog was baked into the signed binary. For external catalogs, we need per-entry verification.

### 3.2 Entry-level signature

The catalog schema (v1.1) gains an optional `signature:` field on each entry:

```yaml
- name: cert-manager-internal
  chart: cert-manager
  repo: https://internal.example.com/charts
  ...
  signature:
    bundle: https://catalogs.example.com/signatures/cert-manager-internal.bundle
```

`signature.bundle` is a Sigstore bundle URL (per the [Sigstore bundle protobuf](https://docs.sigstore.dev/cosign/signing/signing_with_bundles/)).

### 3.3 Load-time verification

When the catalog loader encounters an entry with `signature`:

1. Fetch the bundle.
2. Verify it covers the entry's canonical serialization (deterministic YAML marshal — details in implementation).
3. Extract the signing identity (OIDC issuer + subject).
4. Match against `SHARKO_CATALOG_TRUSTED_IDENTITIES` (see §3.4).
5. Mark the entry `verified: true` (trusted) / `verified: "signature-mismatch"` / `verified: "untrusted-identity"`.

Unsigned entries are accepted but flagged `verified: false`. The UI renders a neutral chip for unsigned and a green "Verified — `<issuer>`" chip for verified. Untrusted or mismatched shows a warning chip with a tooltip.

#### 3.3.1 Sidecar verification entry point

Invoked by Subsystem A's fetcher after a successful YAML pull when a sidecar URL is present. Takes `(catalog-bytes, sidecar-URL, trust-policy)` → returns `(verified-bool, issuer-identity)`. This keeps Subsystem A independent of cosign code: if Subsystem B is not compiled in / not configured, Subsystem A skips the call and every entry in that fetched snapshot inherits `verified: false`. Subsystem B owns the signing package (`internal/catalog/signing/`); Subsystem A only holds an interface.

### 3.4 Trust policy

Env var `SHARKO_CATALOG_TRUSTED_IDENTITIES` — comma-separated regex list mirroring the [cosign `--certificate-identity-regexp`](https://docs.sigstore.dev/cosign/verifying/verify/) pattern.

Default (v1.23 ship):
```
https://github.com/cncf/.*/.github/workflows/.*
https://github.com/MoranWeissman/sharko/.github/workflows/release.yml@refs/tags/v.*
```

Operators extend via env: `SHARKO_CATALOG_TRUSTED_IDENTITIES=<defaults>,https://github.com/myorg/.*/.github/workflows/.*`.

### 3.5 UI

- **Verified badge:** green checkmark chip with issuer identity on hover.
- **Unsigned / unverified:** neutral chip "Unsigned" or warning "Unverified — see signature docs".
- **Filter:** Browse tab's existing `curated_by` multi-select gains a pseudo-tag `signed` that filters to `verified: true` entries.

### 3.6 Release pipeline — sign embedded catalog entries

A new GitHub Action step (reusing the existing `.github/workflows/release.yml` cosign setup — keyless OIDC) produces per-entry signatures for every embedded catalog entry on tag push. The signatures are attached as release assets under `catalog-signatures/` and also materialized into `catalog/addons.yaml` in the release commit so embedded-source entries ship pre-signed.

The verification policy against the embedded catalog still allows `signature: null` (embedded entries inherit trust from the binary signature) — this is for third-party consumers who might re-distribute.

## 4. Subsystem C — Automated trusted-source scanning bot

### 4.1 Goal

Nightly GitHub Action scans upstream curated lists, detects additions / changes, and opens PRs to `catalog/addons.yaml`. **The bot never auto-merges** — human review is required. This keeps curation quality high while eliminating the maintainer chase.

### 4.2 Scope (v1 — infra first, sources grow later)

- **CNCF Landscape YAML:** `https://raw.githubusercontent.com/cncf/landscape/master/landscape.yml` — filter by category (security, observability, networking, storage, …) and maturity (graduated + incubating).
- **AWS EKS Blueprints addon registry:** [`aws-quickstart/cdk-eks-blueprints`](https://github.com/aws-quickstart/cdk-eks-blueprints) — enumerate `lib/addons/*` directories, extract metadata.
- **GKE marketplace / Azure AKS:** deferred to v1.24 — requires authenticated API access + partner-scoped auth the bot doesn't have on a public runner.

### 4.3 Architecture

- **Workflow:** `.github/workflows/catalog-scan.yml` — cron `0 4 * * *` (04:00 UTC daily).
- **Script:** `scripts/catalog-scan.mjs` (Node ESM for consistency with existing `scripts/docs-screenshots.mjs` from v1.22).
- **Token:** uses `GITHUB_TOKEN` for GitHub API; caches upstream responses in the runner's filesystem per-run.

### 4.4 Script logic

1. Fetch upstream source manifests.
2. Normalize each into the Sharko catalog schema shape (name, chart, repo, category, source_url, maintainers, license).
3. Diff against current `catalog/addons.yaml` — detect adds and version-relevant updates.
4. For each change, pre-compute:
   - OpenSSF Scorecard (re-run the v1.21 Scorecard fetch — §4.6 of the v1.21 design).
   - License validity (allow-list check — §4.9 of v1.21).
   - Chart resolvability (same check as the `catalog-validate` CI workflow).
5. Open one PR per scan run with:
   - Branch: `catalog-scan/<YYYY-MM-DD>` (see open question §7 — branch naming / draft policy).
   - Labels: `catalog-scan`, `needs-review`.
   - Body: markdown table with each proposed change + Scorecard + license + resolvability signals.
   - No commit if no changes detected.

### 4.5 Rate limits + safety

- Exponential backoff on GitHub API 429 / 5xx.
- Upstream response caching per-run (same scan should not double-fetch).
- **No cosign step in the bot.** Per-entry signing happens on release publish (Subsystem B §3.6), not in the scanner. The bot's job is curation signals; the release cuts the authoritative signed catalog.

### 4.6 Reviewer UX

The PR body is the primary reviewer surface. Each proposed entry shows:

```
| Addon | Action | Scorecard | License | Chart resolvable | CNCF tier |
|-------|--------|-----------|---------|------------------|-----------|
| Kyverno | add | 8.7 (Strong) | Apache-2.0 | OK | graduated |
```

Reviewer either merges as-is, or commits fixes on the branch before merge.

## 5. Backward compatibility

- **No breaking changes.** All three subsystems are opt-in.
- **Embedded catalog** remains the baseline; private catalogs + per-entry signatures are additive layers.
- **Schema bump to v1.1** — adds optional `signature:` field. v1.0 catalogs (and Sharko v1.22 or earlier binaries reading v1.1 YAML) load fine since unknown fields are tolerated (per v1.21 §4.2 "Unknown fields are tolerated").
- No changes to `POST /api/v1/addons` (the "add addon to user's catalog" Operation 1 flow stays identical).
- No changes to Operation 2 (deploy flow).

## 6. Out of scope (v1.23)

- **UI editor for building private catalogs** — defer to v1.24+. For v1.23, third-party catalogs are produced outside Sharko (hand-edited YAML, or generated by another tool) and served over HTTPS.
- **Multi-source conflict resolution beyond "embedded wins"** — if two third-party catalogs define the same `name`, resolution is alphabetical on URL order + a surfaced conflict note. Richer policies (e.g., priority lists, namespacing) defer.
- **GKE marketplace / Azure AKS scanning** — defer to v1.24 (API auth complexity on public runners).
- **Air-gap workflow for signature verification** — defer. v1.23 requires outbound HTTPS to the Sigstore transparency log (Rekor) for verification. Air-gapped operators skip Subsystem B and rely on Subsystem A with trusted network-level provenance.
- **Catalog entry "rollback"** — if a scanned update introduces a regression post-merge, rollback is a manual revert PR; no automated rollback in v1.23.
- **RBAC on "which third-party catalogs a user can see"** — covered by V2.x scoped RBAC roadmap; not v1.23.

## 7. Open questions (to resolve during v1.23 execution)

1. **`SHARKO_CATALOG_URLS` persistence:** **Resolved: env-only** per stateless principle (NFR-V1.21 + this doc §2.7). Story V123-1.1 implements the parser + SSRF guard in `internal/config/catalog_sources.go`; the Settings page surfaces a read-only view only. ConfigMap persistence is explicitly out of scope for v1.23 — revisit in v1.24 if operators push back.
2. **Per-entry signature refresh cadence:** re-verify on every catalog reload (every 1 h for third-party sources), or only on fetch? Performance vs freshness. Leaning "on fetch" — re-verify only when the catalog YAML bytes change; cache the verification result.
3. **Scanning bot PR target:** open draft PRs to `main`, or to a persistent `catalog-updates` branch that a human rebases periodically? Draft-to-main = more noise but each PR is independent; `catalog-updates` = cleaner but reviewer has to manage a long-running branch. Leaning draft-to-main with the `catalog-scan` label.

Questions 2 and 3 resolve during v1.23 story execution (V123-2.2 for #2, V123-3.4 for #3). Question 1 is resolved above.

## 8. CNCF alignment (incremental lift)

v1.21 lifted the incubation self-assessment rubric ~40% → ~50%. v1.23 adds:

| Dimension | v1.23 delivers | Rubric lift |
|---|---|---|
| **Supply chain — third-party provenance** | Per-entry cosign + trust policy | Significant — covers the last "how do you trust non-first-party content" gap |
| **Supply chain — automated freshness** | Scanning bot + CNCF Landscape integration | Moderate — shows the catalog is kept fresh without maintainer heroics |
| **Governance — extensibility** | Third-party catalogs with source attribution | Moderate — proves the project doesn't lock users into the vendor's curation |

Estimated rubric lift: ~50% → ~60%. Closing the remaining gap (SSO, HA, threat model) is the V2 hardening epic.
