# Addons Repository Structure

The addons repository is a Git repository that Sharko manages. It contains the ArgoCD ApplicationSet and all addon Helm values files. ArgoCD watches this repo and deploys addons based on its contents.

## Directory Layout

```
your-addons-repo/
  appset/
    appset.yaml              # ArgoCD ApplicationSet (generated by sharko init)
  addons/
    cert-manager/
      values.yaml            # base values for all clusters
      clusters/
        prod-eu.yaml         # per-cluster overrides for prod-eu
        staging.yaml         # per-cluster overrides for staging
    monitoring/
      values.yaml
      clusters/
        prod-eu.yaml
    ingress-nginx/
      values.yaml
      clusters/
```

## The ApplicationSet

The `appset/appset.yaml` file is the engine that drives all addon deployments. It is a single ArgoCD ApplicationSet that generates one ArgoCD Application per (cluster, addon) pair.

Sharko generates this file during `sharko init` and never modifies it after that — it is a static template. To change the ApplicationSet (e.g., add new parameters), edit it directly and commit.

## Base Values (`values.yaml`)

Each addon has a `values.yaml` in its directory. This file contains the default Helm values applied to every cluster that has the addon enabled.

Example `addons/cert-manager/values.yaml`:

```yaml
installCRDs: true
replicaCount: 1
resources:
  requests:
    cpu: 10m
    memory: 32Mi
```

## Per-Cluster Overrides

The `clusters/` subdirectory contains per-cluster values files. Sharko creates these files when:

- A cluster is registered with the addon enabled
- A per-cluster upgrade is performed
- A per-cluster values change is made via the UI

The filename is `<cluster-name>.yaml`. Values in this file are merged on top of `values.yaml` (Helm values merge semantics).

Example `addons/cert-manager/clusters/prod-eu.yaml`:

```yaml
replicaCount: 2
resources:
  requests:
    cpu: 50m
    memory: 64Mi
```

## How ArgoCD Uses This Structure

The ApplicationSet iterates over:

1. The list of addons in `addons/` (each subdirectory is an addon)
2. The list of clusters (from the per-cluster override files in `clusters/`)

For each (cluster, addon) pair, it creates an ArgoCD Application that:

- Installs the addon's Helm chart at the version specified in the ApplicationSet parameters
- Merges `values.yaml` and `clusters/<cluster>.yaml` as Helm value files
- Targets the remote cluster

## Version Tracking

Addon versions are stored in the ApplicationSet's parameter list, not in the values files. When `sharko upgrade-addon` runs, it updates the version in the ApplicationSet and opens a PR.

This means:

- The ApplicationSet is the version source of truth
- Drift is detected by comparing the ApplicationSet version against what ArgoCD reports as deployed
- All clusters sharing an addon use the same ApplicationSet version unless per-cluster overrides exist

## Sharko's Git Operations

All Git operations go through a mutex (one at a time, no concurrent PRs):

| Operation | Git Action |
|-----------|-----------|
| `sharko init` | Creates `appset/` and `addons/` structure, opens PR |
| `sharko add-addon` | Creates `addons/<name>/values.yaml`, opens PR |
| `sharko add-cluster` | Creates `addons/<addon>/clusters/<cluster>.yaml` for each addon, opens PR |
| `sharko remove-cluster` | Deletes all `clusters/<cluster>.yaml` files, opens PR |
| `sharko upgrade-addon` | Updates version in ApplicationSet, opens PR |
| `sharko remove-addon` | Deletes `addons/<name>/` directory, opens PR |
