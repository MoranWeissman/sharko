# Smart values layer (v1.21)

When you add an addon through the [marketplace](marketplace.md) (or any path that reaches `POST /api/v1/addons`), Sharko generates a values file for you based on the chart's upstream `values.yaml`. The generated file is annotated, split into global vs per-cluster sections, and stamped with a self-describing header so Sharko knows when to ask you to refresh it.

This page explains what the smart-values layer does, what fields it considers cluster-specific, and how it interacts with the per-cluster overrides editor.

## What the layer produces

For each addon you add, Sharko writes one file at `configuration/addons-global-values/<addon>.yaml`. The file has three sections, in order:

```yaml
# Generated by Sharko from cert-manager@v1.14.4 on 2026-04-19
# Chart source: https://charts.jetstack.io
# AI annotation: disabled
# sharko: managed=true

# --- the global values body ---
cert-manager:
  installCRDs: true
  ingress:
    enabled: true
    # host: <cluster-specific>
  # replicaCount: <cluster-specific>
  resources:
    # requests: <cluster-specific>
  serviceAccount:
    create: true

# --- per-cluster overrides template ---
# Copy under the addon's stanza in configuration/addons-clusters-values/<cluster>.yaml.
# cert-manager:
#   "ingress.host": <set per cluster>
#   replicaCount: <set per cluster>
#   "resources.requests.cpu": <set per cluster>
```

### The header

Four lines, in this order:

| Line | Meaning |
|------|---------|
| `# Generated by Sharko from <chart>@<version> on <YYYY-MM-DD>` | The version of the upstream chart this file was generated from. The version-mismatch banner compares this against the catalog. |
| `# Chart source: <repo URL>` | The Helm repo Sharko fetched the upstream `values.yaml` from. |
| `# AI annotation: enabled\|disabled` | Whether the file was annotated with AI-generated inline comments. v1.21 ships with annotation disabled by default; the AI annotator lands in v1.21+. |
| `# sharko: managed=true` | The marker that opts this file into Sharko-managed lifecycle features (the version-mismatch banner, the refresh action). Hand-edit the line out and Sharko will leave the file alone. |

An optional fifth line, `# sharko: ai-annotate=off`, opts the addon out of AI annotation when the global toggle is on (planned for v1.21+).

### The body

The chart's `values.yaml` is wrapped under the addon name (matches v1.20's convention). Comments, ordering and formatting from upstream are preserved verbatim. Cluster-specific scalar values are replaced with a commented placeholder `# <key>: <cluster-specific>` at their original position so you can see they exist and know they need a per-cluster value.

### The per-cluster template

A fully commented YAML stanza at the bottom of the file. It lists every cluster-specific field detected, in flat dotted-key form, ready to be uncommented and pasted into a cluster's overrides file at `configuration/addons-clusters-values/<cluster>.yaml`.

You don't usually need to copy it by hand — see [Per-cluster template seeding](#per-cluster-template-seeding) below.

## Cluster-specific field detection

Sharko ships a heuristic list of patterns that identify cluster-specific fields. The list lives at `internal/orchestrator/smart_values.go` (`clusterSpecificPatterns`); a schema-PR can extend it.

The patterns Sharko ships in v1.21:

- Connection / ingress targets: `*.host`, `*.hostname`, `*.domain`, `*.endpoint`, `*.url`
- TLS / ingress structure: `*.ingress.*`, `*.ingress`, `*.tls.*`, `*.tls`
- Sizing: `*.replicaCount`, `*.replicas`
- Resource sizing: `*.resources.*`, `*.resources`
- Storage: `*.persistence.*`, `*.persistence`, `*.storageClass`, `*.storageClassName`, `*.pvc.*`, `*.pvc`
- Scheduling: `*.nodeSelector`, `*.tolerations`, `*.affinity*`
- Secrets / external integrations: `*.externalSecret*`, `*.existingSecret`, `*.secretName`
- Cluster identity: `*.clusterName`, `*.region`, `*.environment`
- IRSA / Workload Identity: `*.serviceAccount.annotations.*`

Pattern semantics:

- Matching is case-insensitive against the **full dotted path** of the field.
- A leading `*` matches zero or more path segments. So `*.host` matches both `host` (at the root) and `service.host` (nested).
- A trailing `*` matches zero or more path segments. So `*.ingress.*` matches `ingress`, `ingress.tls`, and `controller.ingress.tls.hosts`.

When the pattern doesn't catch a field that's actually cluster-specific (or, conversely, flags a global field), open a PR against `internal/orchestrator/smart_values.go`. The list is intentionally conservative — false positives are recoverable (the user uncomments the placeholder), but false negatives leak cluster-shape into the global file.

The optional **AI annotation** layer (planned for the v1.21 series) adds an LLM call that finds non-obvious cluster-specific fields and unions them into this set. AI is additive only — it never removes paths the heuristic flagged.

## Per-cluster template seeding

When you enable an existing addon on a **new** cluster (via the cluster page → Enable Addon), Sharko reads the per-cluster template block from the global values file and seeds the cluster's overrides file (`configuration/addons-clusters-values/<cluster>.yaml`) with the same fields, uncommented, under the addon's stanza:

```yaml
# Before — cluster file has only other addons
external-secrets:
  enabled: true

# After — Sharko added the cert-manager stanza with template fields
cert-manager:
  enabled: true
  ingress:
    host: <set per cluster>
  replicaCount: <set per cluster>
  resources:
    requests:
      cpu: <set per cluster>
external-secrets:
  enabled: true
```

Three rules govern the seeder:

1. **Other addons are never touched.** Only the addon's own stanza is added.
2. **Existing user fields are never overwritten.** If the cluster file already has `cert-manager.ingress.host: my-real-host`, the seeder leaves it alone. The seeder only fills in keys that are absent.
3. **The seeder only fires on the initial enable.** If the cluster file already has a `cert-manager:` stanza with non-`enabled` fields, the seeder treats it as configured and skips entirely.

The template block in the global values file is **never re-applied to existing cluster files** when the global file is regenerated (e.g. after a Refresh from upstream). Per-cluster overrides are user-owned territory once they exist.

## When the values file gets regenerated

Three triggers regenerate the global values file:

1. **Add Addon.** When you add a new addon through the marketplace (or the API), Sharko fetches the chart's upstream `values.yaml` for the version you picked, runs the smart-values pipeline, and writes the file as part of the same PR that adds the entry to `addons-catalog.yaml`. If the chart is unreachable at add time, Sharko falls back to a minimal `<name>:\n  enabled: false` stub — you can run a Refresh later once connectivity is restored.

2. **Refresh now (banner).** When the catalog version is ahead of the values-file version, the Values tab shows a banner with a Refresh now button. Clicking it regenerates the global file at the catalog version. See [Refreshing values from upstream](values-editing.md#refreshing-values-from-upstream-v121).

3. **Re-add (idempotency).** Sharko's add path skips re-writing the global values file if a file already exists for that addon. This protects hand edits if you ever delete and re-add a catalog entry.

Manual edits via the Values tab editor land directly through the existing Tier 2 PUT endpoint — they don't go through the smart-values regenerator.

## Backwards compatibility with v1.20 files

Values files written before v1.21 don't have a `# sharko: managed=true` header. They keep working unchanged: the version-mismatch banner stays suppressed (no header to compare against), and the editor renders the file as-is. To opt a legacy file into managed lifecycle, click **Refresh now** on the chart's Values tab once you've confirmed the upstream chart hasn't drifted; that will regenerate the file with the v1.21 header.

## API surface

The smart-values layer is wired into two existing endpoints, no new routes:

- `POST /api/v1/addons` — when called, Sharko pre-fetches `values.yaml` for the requested chart/version and runs the smart-values pipeline. The file is committed in the same PR that adds the catalog entry.
- `PUT /api/v1/addons/{name}/values` with `{"refresh_from_upstream": true}` — replaces the global values file with a freshly-regenerated one for the catalog-pinned chart version.

The previous `POST /api/v1/addons/{name}/values/pull-upstream` endpoint that v1.20.1 shipped was removed in v1.21.
